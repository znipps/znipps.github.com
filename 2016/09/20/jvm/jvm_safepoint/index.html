JVM之Safepoint

介绍虚拟机中safepoint的重要性，以及相关的设计实现！

概述

Java虚拟机中有一个重要的概念，就是safepoint，从表面的名字意义上，我们可以知道，这个是关于线程安全的。safepoint在虚拟机中很重要，特别是gc中。它保证gc的语义实现。我们先介绍几本概念，然后一个个的介绍每一部分的设计。
safepoint在gc中很常见，也是stop-the-world的实现机制。进入safepoint，虚拟机保证只有VMThread线程运行，其他线程挂起，也就是说Java虚拟机的GC是通过VMThread作为主要进入点实现的。在Java虚拟机执行的指令有以下几个部分：

  • 解释器
    Java虚拟机是解释器执行，虚拟机采用了模板解释器解释虚拟机执行的执行，那么在进入安全点，这个是比不可少的点，通过将解释器有_active_table修改为_safept_table实现。
  • 编译代码
    Java虚拟机有一个版本叫hotspot，为了运行效率，虚拟机根据热点算法将热点的部分翻译成本地代码，那么从本地代码如何进入safepoint，也是需要考虑的部分。
  • 本地代码
    Java虚拟机支持调用本地代码，那么如何从本地代码进入safepoint同样也是需要考虑的一部分。
  • 线程处于阻塞状态
    Java虚拟机不允许唤醒虚拟机处于safepoint时唤醒处于Block状态的线程
  • 线程处于VM状态
    Java虚拟机同样不允许虚拟机切换状态

VMThread线程需要和其他线程连在一起看，我们重点关注VMThread和JavaThread线程。虚拟机分配对象的时候发现内存空间不足够，虚拟机向VMThread线程发送执行垃圾回收的指令，VMThread执行进入safepoint操作，一般会涉及以下几个操作：

  1. VMThread获取Threads_lock锁，让虚拟机不在接受创建线程工作
  2. 统计当前的线程数量，此时线程数量不会增减，由于持有锁，不会创建，也不会减少线程数量
  3. 将解释器的指令跳转表由_active_table修改为_safept_table
  4. 检查当前运行的线程数量，由于第三部，其他线程执行字节码的时候，会将其他线程挂起，所以还在运行的线程会越来越少
  5. 由于进入safepoint的线程和阻塞的线程数量一致，所以还需要检查有多少线程进入了阻塞状态，在检查的过程中，VMThread会暂时释放Safepoint_lock锁，进入wait状态
  6. 检查所有的线程都进入了阻塞状态表示当前虚拟机的线程除了VMThread线程在运行，其他的线程都处于阻塞状态了。此时VM进入了safepoint状态
  7. VM通过了所有的safepoint操作后，退出safepoint，还原字节码转换表，释放Threads_lock，让VM进入正常运行。

对于JavaThread线程执行的操作，一般会涉及以下几步：

  1. JavaThread执行字节码跳转表_safept_table将执行线程状态转换，而线程状态转换会检查safepoint状态
  2. 变更当前线程状态，让其不在RUNNING，线程修改memory_serialize的内存内容
  3. 由于此时处于safepoint状态，所以执行线程获取Safepoint_lock锁，由于VMThrea会释放此锁，所以每个线程都有机会运行，减少当前还在还需要阻塞的数量。如果当前线程发现还需要阻塞数量为0，那么通知VMThread继续运行。
  4. 前面的工作做好，JavaThread此时获取Threads_lock，由于此时Threads_lock已经被VMThread持有，所以当前线程处于阻塞状态
  5. VMThread执行完所有的safepoint操作，会释放Threads_lock，此时线程会继续执行并获取Threads_lock锁，将线程状态还原
  6. 释放Threads_lock锁，由于此时VMThread退出safepoint是会将字节码跳转表还原，这样线程就继续之前的状态运行了。

从上面的流程中，我们看到了VM进入safepoint状态的描述过程，后面详细描述各个状态进入safepoint的操作过程，在这个场景中，我们可以看到锁的利用很重要，通过规定锁的顺序保证了避免死锁，线程规定Safepoint_lock->Thread_lock。所以在使用锁的时候避免出现死锁，线程顺序可以避免死锁。

JIT进入Safepoint

JIT编译的本地代码，JIT会在以下几个点插入指令，由于探测线程是否应该进入Safepoint，虚拟机需要进行垃圾回收，有一个状态会进行stop-the-world，让所有的线程挂起，为了防止进入safepoint的时间太长，导致无法安全进入block状态，hotspot会把编译成本地代码的循环体插入test指令，这个章节详细描述这个过程，从编译器编译java字节码插入test指令，以及怎么进入block过程：

  1. 循环的末尾 (防止大循环的时候一直不进入safepoint，而其他线程在等待它进入safepoint)
  2. 方法返回前
  3. 调用方法的call之后
  4. 异常抛出
    以下章节分别详细描述以上几种情况的过程，以上几种情况遵守以下流程

循环进入SafePoint

虚拟机需要进行垃圾回收，为了防止进入safepoint的时间太长，导致无法安全进入block状态，hotspot会把编译成本地代码的循环体插入test指令。这个指令不会修改任何寄存器，但是会读内存单元，但此时safepoint的内存单元已经被VMThread修改为禁止访问，此时会触发异常处理。CPU会捕获这个异常交给异常处理程序。我们这里详细描述这个过程。

为什么源码注释说Serialization Page的性能比membar还好？

因为原本Java thread和VM thread都要做membar；用了serialization page之后只有VM thread那边要做mprotect+membar（比原本慢），而Java thread只要做一个普通的serialization page read（比原本快）。VM thread只有一个而Java thread可能有很多个。细节参考Asymmetric-Dekker-Synchronization
