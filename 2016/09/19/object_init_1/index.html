Object 初始化过程(一)

备注：文章中所有的平台都是在Linux上，Windows不做考虑。文中只是部分，后续慢慢更新。

前言

Object类初始化前必须完成JVM规范中的加载，链接等。

执行过程分析

在执行初始化的过程中，在内存中表示的instanceKlass都已经建立好了映射关系，instanceKlass已经建立好了相关的数据和控制结构，供运行时使用。在初始化之前，解释器会验证类是否已经链接，保证符合JVM规范。

每一个类或接口都会有一个初始化锁，加载和初始化过程中需要特别注意多线程加载导致的并发问题，在执行初始化过程同样如此，所以首先需要获取初始化锁。

正如所描述，初始化必须在链接之后。
下面就进入了JVM规范中详细描述类初始化的过程，我们一步步来看,我们先看看初始化的整个过程，后面详细描述一些过程：

第一步、同步 Object 的初始化锁 ol。这个操作会导致当前线程一直等待直到可以获得 ol 锁。
第二部、如果 Object 的 Class 对象显示当前 Object 的初始化是由其它线程正在进行，那么当前线程释放
ol 并进入阻塞状态，直到它知道初始化工作已经由其它线程完成，那么当前线程在此重试此步骤。

第三部、如果 Object 的 Class 对象显示 Object 的初始化正由当前线程在做，这就是对初始化的递归请求。
释放 ol 并正常返回。
第四部、如果 Object 的 Class 对象显示 Class 已经被初始化完成，那么什么也不做。释放 ol 并正常
返回。
第五步、如果 Object 的 Class 对象显示它处于一个错误的状态，就不再初始化了。释放 ol 并抛出
NoClassDefFoundError 异常。

第六步、否则，记录下当前线程正在初始化 Object 的 Class 对象，随后释放 ol。根据属性出现在
ClassFile 的顺序，利用常量池中的 ConstantValue 属性（ §4.7.2）来初始化 Object 中的各个
final static 字段。
第七步、接下来，如果 Object 是类而不是接口，而且它的父类 SC 还没有被初始化过，那就对于 SC 也
进行完整的初始化过程。当然如果必要的话，需要先验证和准备 SC。如果在初始化 SC 的时候因
为抛出异常而中断，那么就获取 ol 后将 Object 的 Class 对象标识为错误状态，并通知所有正在等待
的线程，最后释放 ol 并异常退出，抛出与 SC 初始化遇到的异常相同的异常。

第八步、之后，记录一些加载信息，供后续使用。执行 C 的类或接口初始化方法

第九步、如果正常地执行了类或接口的初始化方法，之后就请求获取 ol，标记 Object 的 Class 对象
已经被完全初始化，通知所有正在等待的线程，接着释放 ol，正常地退出整个过程。
第十步、 否则，类或接口的初始化方法就必须抛出一个异常 E 并中断退出。如果 E 不是 Error 或
它的某个子类，那就创建一个新的 ExceptionInInitializerError 实例，然后将此实例作为
E 的参数，之后的步骤就使用 E 这个对象。如果因为 OutOfMemoryError 问题而不能创建
ExceptionInInitializerError 实例， 那在之后就使用 OutOfMemoryError 异常对象作为
E 的参数。
第十一步、 获取 LC，标记下 C 的 Class 对象有错误发生，通知所有的等待线程，释放 LC，将 E 或
上述的具体错误对象作为此次意外中断的原因。

以上，按照JVM规范对Object初始化的一个翻译过程，Object的初始化必须满足JVM规范的要求。
